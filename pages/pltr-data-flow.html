數據唔係直接讀 DB — 中間有一層

Palantir Ontology 嘅核心設計：應用層永遠唔直接打 Backing Dataset。所有讀寫經過一個 search-optimized 嘅 Index Layer（Object Database），令查詢毫秒級回應，user edit 即時可見。

好似 JPA 嘅 L2 Cache — 應用層經過一層優化過嘅 index 讀寫，唔係每次都打底層 DB。

三層分離：Schema / Raw Data / Index

PLTR Ontology 唔係一個 database。佢有 3 層完全分離嘅組件，各自獨立演進。

OMS（Schema 層）

Schema 定義住呢度 — Object Types、Link Types、Action Types。用戶喺 Ontology Manager UI 配置，唔使寫 code。

Java 類比：JPA @Entity metamodel，但全平台共用、UI 配置

Backing Dataset（Raw Data）

原始數據嘅真正儲存位置。可以係 Parquet in S3、PostgreSQL table、Kafka stream。Ontology 唔擁有數據 — 佢只係映射。

Java 類比：JPA @Table mapping 到現有 DB table

Object Database（Index）

全部 query 打呢度。Inverted index，只 index 標記為 Searchable 嘅 properties。毫秒級回應。

Java 類比：Elasticsearch index 搭喺 PostgreSQL 上面

寫入路徑 A：Funnel Pipeline — 數據點樣入 Index

原始數據到可查詢 object 嘅 4 步 incremental 轉換。唔係每次 full reindex — 只處理上次以嚟嘅 diff。

Java 類比Funnel 等價
EntityManager.persist() + DB transactionBacking Dataset transaction
Hibernate L2 cache invalidationChangelog Job（detect diff）
@Indexed annotation on fieldsSearchable property hint
Elasticsearch index refreshHydration to search nodes

Incremental = git diff

Funnel 係 incremental 嘅 — 唔係每次 full reindex。只處理上次以嚟嘅 diff。好似 git diff — 唔使每次 clone 成個 repo。10 億 rows 只改 1000 行 → 只 index 呢 1000 行。

讀取流程：App → Ontology API → Index → 結果

應用程式查詢從來唔接觸 Backing Dataset。全部讀取經 Object Database 嘅 inverted index。

GET /objects/Task?status=FAILED Object Database 查 status index: FAILED → [TSK-005] → 毫秒級回應，唔使 scan 全部 rows

Searchable Render Hint

唔係每個 property 都會被 index。每個 property 有 Searchable 設定：ON → 建 inverted index，可以 search / filter；OFF → 唔建 index，慳 reindexing 時間。Java 類比：@Indexed annotation 控制邊啲 column 有 DB index。

寫入路徑 B：Actions — 用戶操作點樣即時生效

User edit 唔會寫回 Backing Dataset。走獨立嘅 queue path，改動即時可見。

Query-Time Merge 機制

Offset 係一個順序號碼。Base Index 記住自己 index 到邊個 offset。Query 時套用後續 edits：

Base Index（indexed at offset 0）: TSK-005.status = "IN_PROGRESS" Edit Queue: offset 1: TSK-005.status = "FAILED" offset 2: TSK-005.status = "RETRY", robotId = "R-HIK-01" Query "status=IN_PROGRESS": Step 1: Base Index → [TSK-002, TSK-005] Step 2: Apply offset 1 → TSK-005 改成 FAILED（移除） Apply offset 2 → TSK-005 改成 RETRY（仍然唔係 IN_PROGRESS） Result: [TSK-002]

六大微服務：完整架構

PLTR Ontology 後端由 6 個獨立服務組成，各司其職。

1 OMS

Schema 定義 — Object Types / Link Types / Action Types。全平台嘅「型態系統」。

2 Object Data Funnel

ETL pipeline：Changelog → Merge → Index → Hydrate。將 raw data 轉成 searchable index。

3 Object Databases (OSv2)

Indexed data storage + query engine。所有 query 嘅最終目的地。

4 Object Set Service (OSS)

讀取 API：search / filter / aggregate。應用層同 Object Database 之間嘅接口。

5 Actions Service

寫入 API：structured edits + 權限 + validation。User edit 嘅入口。

6 Functions on Objects

自定義業務邏輯。好似 Spring @Service methods，但註冊喺 Ontology 上。

核心洞察

PLTR 嘅架構係為多租戶、read-only 數據源、10 億級數據而設計。我哋 MVP 唔需要呢啲，但保留核心概念（schema-driven、通用引擎、structured actions），等升級時可以自然演進，唔使重寫。

架構參考基於 Palantir 官方文檔。Query-time merge 機制係基於通用 CS pattern 嘅推斷，唔係 PLTR 官方公開嘅實現細節。
