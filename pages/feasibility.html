軟件難度 — 逐層誠實評估

之前講「軟件唔複雜」係錯嘅。以下係逐層拆解。

Layer 1: WMS Adapter + Mapping 中-高

唔係寫個 REST client 咁簡單。本質係 ETL mapping — 同 Informatica 一樣嘅苦工。

  • Partner 工廠可能根本冇 WMS — 可能係 Excel / 微信群 / 手寫單
  • 中國 WMS API 文檔質量參差，可能要直連數據庫
  • 唔止 read — 任務完成後要 write back 更新庫存
  • 認證機制每個 WMS 唔同（OAuth / API key / 微信掃碼）
  • API down / timeout / rate limit / 格式變更

Mapping 工作量（每個 WMS source）：

工作項時間 (estimation)備註
理解 source schema + 攞文檔1-2 週如果文檔差或要簽 NDA，可能更耐
Column mapping（source → ontology）1-2 週~50-100 個 fields，逐個確認語義
轉換邏輯（enum mapping、default、format）1 週zone_name "揀貨區" → PICK 呢類
Write-back adapter（ontology → source WMS）1-2 週比 read 難，要處理 transaction
Error handling + retry + monitoring1 週API down 點算？data mismatch 點算？
同 partner 反覆確認 + 修正1-2 週「呢個 field 代表乜？」嘅來回
合計（每個 WMS source）6-10 週Micro-MVP 只接 1 個 source

呢個同 Informatica mapping 嘅苦工一樣 — 冇任何工具可以跳過「bin_code 應該 map 去 location_code」呢種 domain knowledge。PLTR 用 Pipeline Builder（visual drag-drop）做同一件事，速度差唔多，但 output 係 Ontology Object（帶 links + actions），唔係另一張 table。

Layer 2: Ontology 數據模型 中

唔止係 JPA entities。

  • 需要深入倉儲行業知識 — 你冇，partner 有但佢唔寫 code
  • 溝通鴻溝：partner 講「上架」，你理解嘅可能唔同
  • Edge cases 極多：唔同倉唔同規則、批次管理、效期管理
  • Schema evolution：model 不斷變，migration 要管理

Layer 3: Workflow Engine 非常高 ⚠️

唔係狀態機。係 concurrent job scheduler + resource allocator + item router。整個項目最難嘅部分。

3a. 任務排程 + 並發

  • 並發控制：10 個機器人同時揀 50 張單，邊個做邊張？
  • 資源分配：最近嘅？電量最多嘅？載重合適嘅？— optimization problem
  • 死鎖預防：兩個機器人去同一條通道
  • 異常恢復：機器人中途冇電 / 卡住 / 撞嘢
  • 優先級：急單插隊，已分配嘅任務要唔要重排？
  • 一致性：任務做到一半失敗，庫存數據點 rollback？

3b. 貨物屬性 → Robot Routing（信任 DB 數據）

唔同貨需要唔同 robot。SKU 屬性（重量、尺寸、易碎、溫區）從 WMS DB 取得，workflow engine 根據屬性自動 route：

貨物重量RobotWorkflow
小件輕貨<5kgAMR goods-to-personAMR 搬架→人揀→AMR 搬回
中件標準5-30kgAMR transportAMR 直接搬箱
大件重貨>30kgAGV 叉車叉車搬 pallet
易碎品any人工揀 + robot 搬Robot 搬到人→人揀→robot 送走
冷鏈any冷庫專用 robot限時，唔能久離冷區
異形件any可能人工管狀、袋裝，robot 夾唔住

3c. Order Splitting + 匯合

一張單 3 件唔同類型貨 → 拆 3 個 sub-task → 3 種 robot/人處理 → 匯合打包：

  • 拆單：根據 DB 嘅 SKU 屬性自動拆成唔同 robot 嘅 sub-task
  • 匯合同步：3 個 sub-task 完成時間唔同，等齊先打包？先到先包？
  • 部分失敗：1 個 sub-task 失敗，其他 2 個點處理？

Spring State Machine 搞唔掂呢個。實質係建 mini Kubernetes scheduler。

Layer 4: Robot API 非常高 ⚠️

唔係 HTTP call。係 real-time, safety-critical 系統。

  • 中國機器人 API 可能係 proprietary protocol / TCP / ROS / gRPC
  • Persistent connection，唔係 request-response
  • 管理倉庫地圖：上傳 layout、定義路線、充電站
  • 實時狀態 tracking：位置、電量、任務進度
  • 命令確認鏈：send → 收到? → 開始? → 完成?
  • Multi-robot 交通管理：防碰撞、排隊、讓路
  • 安全：send 錯指令 = 機器人撞人/撞貨
  • 文檔可能只有中文、要簽 NDA、要用 Windows SDK

Layer 5: 客戶可配置 Workflow 高 ⚠️

如果 workflow hardcoded，每個客戶都要改 code。必須做成 configurable。

  • 規則衝突：兩條規則 match 同一件貨，用邊條？
  • 規則遺漏：新情況冇 match 到任何規則
  • 規則膨脹：越加越多，冇人記得點解加嘅
  • 學習成本：倉管唔係程式員，IF/THEN 已經難

AI-Driven 方案：

Level 1 智能默認中-高後端邏輯中等，但 config UI 係隱藏成本：forms、validation、preview、override — vibe coding 幫到 UI 組件但幫唔到 UX 設計同業務 edge case
Level 2 自然語言高「大件用叉車，小件用 AMR」→ AI 生成規則
Level 3 學習模式非常高觀察人工操作 → 自動學習 routing（需大量數據）

誠實面對：Micro-MVP 同 product-overview 嘅願景差距好大。願景係 multi-WMS + multi-robot + AI，Micro-MVP 係 1 WMS + 1 robot + sequential。但 micro-MVP 驗證嘅係最核心嘅假設 — adapter 能唔能接入 WMS，能唔能控制機器人。

建議路徑

Phase 0 Kill or Continue（Week 1-2）

確認 partner 工廠用咩品牌機器人。攞機器人 API 文檔。確認現有 WMS / 管理方式。了解貨物類型（重量、尺寸、易碎、冷鏈）。

如果冇 API 或 partner 唔願意投入時間 → 停止，唔好浪費時間。

Phase 1 數據層 + Dashboard（Month 1-4）

核心工作係 WMS mapping（~6-10 週）：理解 source schema → column mapping → 轉換邏輯 → write-back → error handling。同 Informatica mapping 一樣嘅苦工。之後建 ontology 模型 + read-only dashboard。唔碰機器人。

Phase 2 1 台機器人 + 1 種貨 + 1 個任務（Month 5-9）

只控制 1 台機器人、只處理 1 種貨物類型、做 1 種任務（例如：標準箱 A→B 搬運）。Sequential execution，hardcoded workflow。Partner 現場測試。

技術驗證點 — 如果做唔到，後面全部唔使諗。

Phase 3 多貨物類型 + 配置化（Month 10-14）

加入 item routing（唔同重量/尺寸 → 唔同 workflow）。將 hardcoded workflow 抽象為 Level 1 智能默認配置。考慮搵第二個開發者。

Phase 4 Multi-robot + AI 配置（Month 15+）

Multi-robot orchestration、交通管理、自然語言配置 (Level 2)。呢個階段需要團隊，solo dev 唔夠。

最大風險

最大風險唔係你嘅 coding 能力。

1. 機器人 API 質量 你控制唔到

如果機器人廠商冇開放 API、文檔差、或者需要 Windows SDK → 你做唔到 remote 開發。

2. Partner 配合度 軟性風險

Partner 要出人出時間：攞 API 文檔、安排測試、回報 bug、錄影現場。如果佢太忙 → 你卡死。

3. 行業知識溝通 隱性成本

你唔識倉儲，佢唔寫 code。「上架」呢個詞你哋可能有唔同理解。溝通成本會比預期高好多。
