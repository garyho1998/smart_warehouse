↻ Workflow Engine（全流程）

四個核心流程，每個都生成 Task → 分配 Robot → 監控完成。

1

收貨 Receiving

供應商送貨 → 系統從 WMS 收到入庫 Order → 質檢 → 生成上架 Task

InboundOrder → QualityCheck → Task(type=PUTAWAY)
2

上架 Putaway

Scheduler 分配 Task 俾最近嘅空閒 Robot → Robot 搬 Pallet 到目標 Location → 確認入庫 → 更新 Inventory

Task → Scheduler.assign(robot) → Robot.move(from, to) → Inventory.update()
3

揀貨 Picking（Goods-to-Person）

出庫 Order → 計算最優揀貨路徑 → Robot 將貨架/料箱送到揀貨站 → 人工揀選 → Robot 送回

OutboundOrder → PickPlan → Task(type=PICK) → Robot.deliver(shelf, station) → Human.pick() → Robot.return(shelf)
4

出庫 Shipping

揀貨完成 → 包裝 → 分揀到對應快遞 → Robot 搬到出貨口 → 通知 WMS 出庫完成

PickComplete → Pack → Task(type=SORT) → Robot.move(to=dock) → WmsAdapter.confirmShipment()

⚙ Robot 抽象層

統一接口對接任何機器人品牌。上層業務邏輯唔知底層係海康定極智嘉。

public interface RobotAdapter {

  // 發任務俾機器人
  TaskResult dispatchTask(MoveTask task);

  // 查機器人狀態（位置、電量、當前任務）
  RobotStatus getStatus(String robotId);

  // 取消進行中嘅任務
  void cancelTask(String taskId);

  // 取得所有機器人狀態（Fleet 管理）
  List<RobotStatus> getRobotFleet();

  // 狀態變更回調（機器人完成/失敗時通知）
  void onStatusChange(StatusCallback callback);
}

★ 任務排程引擎

唔止係「發任務俾機器人」— 要考慮優先級、負載均衡、異常處理。

⚖ 優先級計算

Score = 訂單緊急度 × 距離權重 × 機器人電量

緊急出庫 > 普通揀貨 > 補貨上架
近距離機器人優先 > 遠距離
電量 >30% 先分配，<20% 自動回充電站

⚖ 負載均衡

唔好所有任務俾同一台機器人。

追蹤每台 Robot 嘅 pending task 數量
Round-robin + 距離優化
避免「一台忙死，其他閒置」

⚖ 異常處理

機器人故障 → 自動重新分配 Task 俾另一台
路徑阻塞 → 通知調度重算路徑
任務超時 → 告警 + 自動 escalate
電量不足 → 暫停接單 + 導航到充電站

! 點解骨幹最難

維度WMS AdapterOntology 骨幹機器人整合
做咩 欄位映射 + API 調用 數據模型 + 業務流程 + 排程 廠商 API + 狀態監控
複雜度 低 — 主要係翻譯 高 — 要理解倉庫全流程 高 — 要處理實時 + 異常
改動頻率 低（API 穩定後好少改） 高（業務需求持續演進） 中（廠商 API 偶爾更新）
出錯影響 數據同步延遲 整個流程停擺 機器人亂走（安全風險）
測試難度 低 — mock API 中 — 需要完整流程測試 極高 — 需要實體機器人
估計時間 3 週 6 週 8 週（含現場調試）

核心洞察

Adapter 代碼最簡單 — 就係欄位映射。
Ontology 骨幹最難 — 因為佢定義咗「倉庫點樣運作」，包括流程、規則、異常處理。改 Adapter 唔影響業務，改骨幹影響所有嘢。
機器人整合最耗時 — 因為真實環境測試需要大量時間（路徑優化、避障、安全、電量管理）。
