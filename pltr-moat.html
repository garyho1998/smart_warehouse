唔係快，係越嚟越快

Ontology 複利效應

傳統 SI 每個客戶都係從零開始。Palantir 每個新客戶都令下一個更快 — 因為 Ontology 累積可複用。

1 Ontology 跨客戶複用

第 1 個物流客戶建 ontology 要 3 個月，第 10 個只需 2 週。因為 80% object types（Shipment, Warehouse, SKU）已存在。每個新客戶豐富模型，令下一個更快。

每次從零建模 → 累積複用，指數加速

2 Pipeline 模板化

Foundry pipeline 係 ontology-aware。"解析 SAP 出庫單" pipeline 直接畀下個 SAP 客戶用，唔使重寫 ETL。

每個客戶寫定制 ETL → ontology-aware pipeline 直接複用

3 Action 可組合

"優化庫存分配" action 基於 ontology 運作，唔理底層係 SAP 定 Oracle 定 CSV。跨客戶即插即用。

每個客戶寫定制邏輯 → 基於 ontology，底層無關

4 SDK 自動生成

Ontology 一改，TypeScript / Python SDK 自動 regenerate，前端自動適配。零手動 API 對接。

改 DB → 改 API → 改前端 → 改文檔 → 改 Ontology → 全棧自動更新

5 Workshop 零代碼建應用

Ontology 定義好 object + relation + action 之後，Workshop 拖拉幾小時出應用。唔使寫 API，唔使寫前端，唔使寫權限 — 全部由 ontology 驅動。

寫 API → 寫前端 → 寫權限 (幾週) → 拖拉組合 (幾小時)

實現層面 5 大壁壘

唔係寫唔出類似功能 — 係每一層都係多年工程量，而且要全部一齊 work 先有複利效應。

1. Ontology Runtime ≠ DB Schema

3-5 年 / 50+ 工程師

唔係一個 schema definition。係一個包含語義關係、權限模型、事件系統、action 框架嘅完整 runtime engine。從零建 = 建一個帶 graph semantics + RBAC + event bus 嘅引擎。

OSS 可以替代？ Neo4j 做 graph、Casbin 做 RBAC、Kafka 做 events — 但將三者整合成 ontology-aware runtime，令佢哋互相感知同協作，係核心難度。冇任何 OSS 項目做到呢件事。

2. 增量計算引擎

2-3 年 / 20+ 工程師

Foundry pipeline track dependency graph，只重算改變嘅部分。唔係 Airflow + cache。係類似 build system (Bazel) 嘅 incremental computation，但用喺 data pipeline — 知道邊啲 ontology object 變咗，只重算下游影響。

OSS 可以替代？ Dagster / Prefect 有 basic lineage，但冇 ontology-aware incremental recomputation。佢哋只知道 "Task B depends on Task A"，唔知道 "Warehouse #42 嘅 SKU 變咗，只需重算呢個倉嘅報表"。

3. 跨棧 Type Generation

1-2 年 / 10+ 工程師

Ontology 定義一改，自動生成：TypeScript SDK、Python SDK、REST API、GraphQL、權限規則。需要 compiler-level 嘅 code generation pipeline，理解前端 + 後端 + 權限整個 stack。

OSS 可以替代？ OpenAPI codegen / GraphQL codegen 只做 API 層，唔理解 ontology semantics。佢哋生成嘅係 "HTTP client"，唔係 "type-safe ontology accessor with built-in permissions"。

4. 異構部署 (Apollo)

3-5 年 / 30+ 工程師

同時支援 AWS + Azure + 本地機房 + 軍事氣隙網絡（零互聯網）。唔係 "multi-cloud"。軍事氣隙 = 完全離線、無 container registry、無 DNS、無外部依賴。架構同 cloud-native 根本唔同。

OSS 可以替代？ ArgoCD / Flux 做 cloud GitOps，但完全唔 handle air-gapped。佢哋假設永遠有網絡。Apollo 嘅離線部署需要自研 artifact distribution、local registry、離線 update mechanism。

5. Graph-based RBAC

1-2 年 / 10+ 工程師

唔係 "邊個可以讀邊張 table"。係 "邊個可以對邊個 object 執行邊個 action，基於邊啲 relationship"。

例：「物流經理可以查看佢管理嘅倉庫中嘅所有訂單，但唔可以修改其他部門嘅」— 呢個 query 需要 traverse ontology graph，唔係一個 WHERE clause。

OSS 可以替代？ OPA / Casbin 做 policy engine，但唔理解 ontology relationship graph。佢哋嘅 input 係 flat attributes (role=admin, resource=orders)，唔係 graph traversal (user→manages→warehouse→contains→orders)。
~10-15 年工程量 / 100+ 專業工程師

而且 5 層要全部一齊 work 先有複利效應 — 只建其中 2-3 層，效果接近零。呢個就係護城河。
